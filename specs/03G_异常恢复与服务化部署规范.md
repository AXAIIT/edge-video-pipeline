# 03G_异常恢复与服务化部署规范

## 适用范围

本规范用于定义视频 pipeline 的异常恢复、日志、退出码、systemd 或 Docker 服务化部署规则。

本阶段重点证明 pipeline 不是只能在终端里手动跑一次，而是具备可诊断、可重启、可降级的工程能力。

## 执行原则

下面的错误码、systemd、Docker 和异常注入方式是推荐路径，不是唯一实现。可以替换服务管理方式、日志格式、健康检查或异常注入脚本，但不能降低证据要求：必须保留异常复现命令、失败日志、恢复策略、退出码、服务日志和问题记录。替代路径必须写入对应 `projects/03_video_pipeline/runs/`，并同步更新 `projects/03_video_pipeline/reports/failure_and_fallback.md` 和 `projects/03_video_pipeline/reports/runbook.md`。

## 路径口径

本 spec 使用当前仓库结构：

- 服务测试配置放在 `projects/03_video_pipeline/configs/pipeline/`。
- systemd 或 Docker 文件放在 `projects/03_video_pipeline/scripts/service/`。
- 失败日志放在 `logs/failures/03_video_pipeline/<target>/`。
- 服务 runtime 日志放在 `logs/runtime/03_video_pipeline/<target>/`。
- 降级报告和 runbook 放在 `projects/03_video_pipeline/reports/`。

## 前置条件

- pipeline 已能基本运行。
- 已确定目标运行方式：命令行、systemd 或 Docker。
- 已准备异常测试场景。
- 已准备日志目录：`logs/runtime/03_video_pipeline/<target>/`、`logs/failures/03_video_pipeline/<target>/`。
- 已准备服务配置输出目录：`projects/03_video_pipeline/scripts/service/systemd/` 或 `projects/03_video_pipeline/scripts/service/docker/`。
- run 记录必须对齐 `projects/03_video_pipeline/specs/00_run_record_template.md`。
- 环境基线必须对齐 `projects/03_video_pipeline/specs/00_environment_baseline_template.md`。
- failure log 必须符合 `benchmark/schemas/video_pipeline_failure_schema.yaml`。

## 异常场景

| 场景 | 触发方式 | 期望结果 |
|---|---|---|
| 输入源断流 | 停止摄像头/断开 RTSP/移动文件 | 重连或清晰错误 |
| 模型加载失败 | 改错模型路径 | 明确错误码和日志 |
| 输入尺寸错误 | 改错配置 | 拒绝启动或进入降级 |
| 后端 runtime 失败 | 传入不可用设备或错误 engine | 清晰错误并退出 |
| output 写失败 | 输出目录不可写 | 记录错误并按策略退出 |
| 队列堆积 | 高 FPS 输入或 sleep 注入 | 丢帧或限流 |

## 执行步骤

### 1. 建立 run 记录

新建：

```text
projects/03_video_pipeline/runs/<yyyymmdd>_<target>_pipeline_failure_service_test/run.md
```

### 2. 定义错误码和日志格式

错误码建议：

```text
0 OK
10 INPUT_OPEN_FAILED
11 INPUT_DISCONNECTED
20 MODEL_LOAD_FAILED
21 BACKEND_RUNTIME_FAILED
30 CONFIG_INVALID
40 OUTPUT_FAILED
50 QUEUE_OVERFLOW
60 WATCHDOG_TIMEOUT
```

日志必须包含：

```text
timestamp
run_id
frame_id
stage
level
error_code
message
recovery_action
expected_behavior
actual_behavior
service_status
related_troubleshooting_id
```

### 3. 异常注入测试

命令模板：

```bash
python3 projects/03_video_pipeline/scripts/inject_failure_tests.py \
  --pipeline build/03_video_pipeline_jetson/video_pipeline_app \
  --pipeline-config projects/03_video_pipeline/configs/pipeline/jetson_tensorrt_pipeline.yaml \
  --model-config projects/03_video_pipeline/configs/models/yolo11n.yaml \
  --backend-config projects/03_video_pipeline/configs/boards/jetson_8gb.yaml \
  --stream-config projects/03_video_pipeline/configs/streams/video_fixed_v1.yaml \
  --cases input_open_failed model_missing invalid_shape output_unwritable queue_overflow \
  --output logs/failures/03_video_pipeline/jetson_8gb/<run_id>_failure_injection.jsonl
```

当前仓库口径说明：

- `inject_failure_tests.py` 已能自动执行 Jetson CLI 子集：`input_open_failed`、`model_missing`、`invalid_shape`、`output_unwritable`、`queue_overflow`。
- `input_disconnect` 需要真实 live-source 配合，当前不能用固定视频文件完全模拟，必须另做板端手工 run。
- Jetson 当前正式 `live-source` 口径已经收敛为 `CSI IMX219`。当前 `03_video_pipeline` 已补入 IMX219 所需的 `RG10 raw -> debayer -> white balance` 专用采集链路，因此 `input_disconnect` 应直接对正式摄像头主线补证。
- `systemd_restart` 属于服务化验证，不属于 CLI 注入脚本的自动覆盖范围，必须通过 `systemctl` 和 `journalctl` 单独留证。
- RK3588 五项 CLI 子集使用 `projects/03_video_pipeline/scripts/run/run_rk3588_failure_injection.sh`，模型缺失用例覆盖为不存在的 `.rknn` 路径；测试门槛与 Jetson 一致。
- RK3588 systemd 使用 `edge-video-pipeline-rk3588.service` 和 `test_rk3588_systemd_service.sh`，必须记录 start/restart/stop 和 journal。
- RK3588 `input_disconnect` 仍要求真实 camera/RTSP live source；未选择 live source 时标记 `blocked_until_live_source_selected`，不能用 playlist EOF 代替断流。

异常注入结果必须记录：

```yaml
run_id:
timestamp:
case_id:
input_source_id:
stage:
error_code:
expected_behavior:
actual_behavior:
recovery_action:
max_recovery_time_sec:
reconnect_count:
frame_id_continuity_after_recovery:
drop_frame_reason_after_recovery:
exit_code:
service_status:
log_path:
status:
related_troubleshooting_id:
```

这些字段必须通过 `benchmark/schemas/video_pipeline_failure_schema.yaml` 校验。`max_recovery_time_sec`、`reconnect_count`、`frame_id_continuity_after_recovery` 和 `drop_frame_reason_after_recovery` 是恢复行为的硬性字段；没有恢复动作时也必须写 `null`、`0`、`not_applicable` 或 `unknown`，不能省略字段。

### 4. systemd 服务

示例文件：

```text
projects/03_video_pipeline/scripts/service/systemd/edge-video-pipeline-jetson.service
```

必须记录：

```bash
sudo cp projects/03_video_pipeline/scripts/service/systemd/edge-video-pipeline-jetson.service /etc/systemd/system/
sudo systemctl daemon-reload
sudo systemctl start edge-video-pipeline-jetson
sudo systemctl status edge-video-pipeline-jetson --no-pager
sudo journalctl -u edge-video-pipeline-jetson --since "15 min ago" --no-pager
sudo systemctl restart edge-video-pipeline-jetson
sudo systemctl stop edge-video-pipeline-jetson
```

如果使用当前仓库的标准化入口，也可以直接执行：

```bash
RUN_ID=$(date +%Y%m%d)_jetson_8gb_systemd_service_test \
  bash projects/03_video_pipeline/scripts/service/test_jetson_systemd_service.sh
```

RK3588 对应入口：

```bash
RUN_ID=$(date +%Y%m%d)_rk3588_8gb_systemd_service_test \
  bash projects/03_video_pipeline/scripts/service/test_rk3588_systemd_service.sh
```

该脚本会统一产出：

```text
logs/runtime/03_video_pipeline/jetson_8gb/<run_id>_systemd_status.log
logs/runtime/03_video_pipeline/jetson_8gb/<run_id>_journal.log
```

run 中必须记录：

```yaml
service:
  service_start_status:
  service_restart_status:
  service_stop_status:
  journal_log_path:
```

### 5. Docker 可选

如果使用 Docker，必须记录：

```bash
docker build -t edge-video-pipeline:dev -f projects/03_video_pipeline/scripts/service/docker/Dockerfile .
docker run --rm --device <device> --network host edge-video-pipeline:dev
```

## 记录要求

- 异常注入命令写入 run。
- 失败日志保存到 `logs/failures/03_video_pipeline/<target>/`。
- 失败日志必须通过 `benchmark/schemas/video_pipeline_failure_schema.yaml` 校验。
- 服务日志保存到 `logs/runtime/03_video_pipeline/<target>/`。
- systemd 或 Docker 配置保存到 `projects/03_video_pipeline/scripts/service/`。
- 降级策略写入 `projects/03_video_pipeline/reports/failure_and_fallback.md`。
- 复现步骤写入 `projects/03_video_pipeline/reports/runbook.md`。

## 输出文件

| 文件 | 要求 |
|---|---|
| `projects/03_video_pipeline/scripts/service/systemd/*.service` 或 `projects/03_video_pipeline/scripts/service/docker/*` | 服务化配置 |
| `logs/failures/03_video_pipeline/<target>/*failure*.jsonl` | 异常注入结果 |
| `logs/runtime/03_video_pipeline/<target>/*service*.log` | 服务运行日志 |
| `projects/03_video_pipeline/reports/failure_and_fallback.md` | 异常恢复和降级策略 |
| `projects/03_video_pipeline/reports/runbook.md` | 服务启动、停止、排障步骤 |

## 验收标准

- 常见异常有可解释日志。
- 服务可启动、停止、重启。
- 断流或后端失败有明确恢复或退出策略。
- 异常注入结果可复现。
- 断流重连必须记录重连次数、最大恢复时间、恢复后 frame_id 连续性或丢帧原因。
- 降级策略不会隐藏失败，必须写入报告。

## 降级和问题库

下面问题必须进入问题库：

- 异常无法复现。
- 日志不足。
- 服务无法自恢复。
- 退出码不明确。
- 断流后进入假运行状态。
